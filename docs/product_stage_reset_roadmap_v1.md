# 温伴 × 忆光 新阶段路线图 v1

## 0. 为什么要重置阶段

旧阶段更多是按技术模块推进：

- 推理链
- 时间模块
- 事件层
- 事实层
- 语音
- 照片墙
- 数字人

这导致一个问题：

功能单点可以完成，但产品版本边界不清楚，用户体验也不一定成立。

所以从现在开始，阶段不再主要按“技术模块”命名，而按“产品版本”命名。

---

## 1. 当前代码属于哪个版本

当前代码基线统一定义为：

## `Web Prototype v0.9`

### 已有能力

- 对话理解与回复控制
- 时间 / 事件 / 修订 / 事实层基础逻辑
- 文件型后端持久化
- 浏览器语音输入输出 MVP
- 基础照片上传 / 展示 / 挂接能力
- 温伴形象生成 MVP
- 原型级隐私说明

### 当前主要问题

- 首次使用流程不够顺
- 照片系统像工具，不像自然材料流
- 网页结构里“输入、整理、展示”边界不清
- 语音还没进入老人场景验证
- 距离 App 主形态还有明显差距

---

## 2. 新版本总图

后续版本从网页原型到 App，建议按下面推进：

1. `V1.0 Web Alpha`
2. `V1.1 Web Beta`
3. `V1.5 Mobile Shell MVP`
4. `V2.0 App Memory Brain MVP`
5. `V2.5 Proactive Memory Assistant`
6. `V3.0 Multi-Entry Companion System`

下面每个版本都带明确目标、范围、验收和当前代码差距。

---

## 3. V1.0 Web Alpha

### 产品定义

让网页端第一次真正像“可用产品”，而不是功能拼接原型。

### 核心目标

- 把首次使用流程做顺
- 把照片变成自然输入，不是表单上传
- 把聊天、忆光、图谱三者边界理清

### 必做范围

1. 首次使用 onboarding 重构
2. 忆光主输入改为：
   - 说一句
   - 放一张照片
3. 图谱页只保留整理功能，不做主输入
4. 设置页只放：
   - 用户卡片
   - 温伴卡片
   - 少量偏好
5. 去掉开发者视角的提示和干扰 UI

### 不做

- 相册自动扫描
- 复杂图像识别
- App 化
- 多端同步

### 验收标准

- 第一次使用不需要理解系统结构
- 照片主入口不再藏在图谱页
- 聊天页和忆光页职责清楚
- 页面看起来像一个产品，而不是一个后台面板

### 当前代码差距

当前已经做了一部分，但还没完全收口。

这会是最近一轮最优先的网页改造版本。

---

## 4. V1.1 Web Beta

### 产品定义

让网页端第一次具备“照片库 + 聊天触发回忆”的雏形。

### 核心目标

- 让照片不只是被动上传
- 开始形成照片候选理解
- 在聊天中自然召回照片

### 必做范围

1. 新增“照片库”页或“照片墙主视图”
2. 支持：
   - 多张导入
   - 按最近导入浏览
3. 后台做轻量照片候选结构：
   - 候选人物簇
   - 候选时间组
   - 候选事件组
4. 聊天中按上下文轻确认：
   - 这张是不是那次的照片
   - 这位是不是刚刚提到的人
5. 一次只问一个问题

### 不做

- 全自动精准识别
- 强制每张图都整理
- 复杂人工编辑器

### 验收标准

- 用户可以理解“照片库”和“忆光”的区别
- AI 能在聊天中自然拿出一张高相关照片确认
- 上传照片不再等同于立刻整理

### 当前代码差距

当前还只有上传和挂接，缺少真正的“照片库”与“聊天触发式召回”。

---

## 5. V1.5 Mobile Shell MVP

### 产品定义

把网页原型套成最小移动壳，验证手机侧真实交互。

### 核心目标

- 让产品第一次在手机上自然使用
- 打通摄像头、相册、麦克风权限

### 技术建议

优先考虑：

- Capacitor
- React Native WebView 壳
- 或团队熟悉的最轻移动封装

### 必做范围

1. 手机启动页
2. 权限引导：
   - 麦克风
   - 相册
   - 摄像头
3. 拍照 / 选图 / 语音输入走真实设备能力
4. 页面适配手机交互节奏

### 不做

- 完整原生重写
- 复杂离线同步
- 商业化发布准备

### 验收标准

- 在手机上可顺畅完成一次完整使用流程
- 拍照与语音不再像桌面网页 hack
- 交互节奏适合手持场景

### 当前代码差距

当前仍主要是网页视角。

---

## 6. V2.0 App Memory Brain MVP

### 产品定义

第一次让它真正成为“外置大脑 App”。

### 核心目标

- 引入照片库读取
- 引入稳定的长期材料仓
- 让聊天、照片、时间开始真正汇流

### 必做范围

1. 相册授权读取
2. 照片库基础索引
3. 时间信息抽取
4. 照片与聊天回忆做候选关联
5. 人物候选确认
6. 首页出现“今日可回看内容”

### 不做

- 重视觉图谱编辑器
- 完整社交关系网络
- 多设备账号体系

### 验收标准

- 用户不只是在上传照片，而是在用一个会整理照片回忆的系统
- 系统能围绕照片库做自然回忆召回
- 首页和聊天页开始具备“外置大脑”感

### 当前代码差距

这一版是从 Web Prototype 向真正 App 迈出的第一大步。

---

## 7. V2.5 Proactive Memory Assistant

### 产品定义

让温伴从“被动响应”进入“克制主动”。

### 核心目标

- 基于材料主动捞起回忆
- 但不做打扰型机器人

### 必做范围

1. 主动唤起策略引擎
2. 触发依据：
   - 时间
   - 纪念日
   - 重复人物
   - 重复场景
   - 照片簇
3. 通知或卡片式主动提醒
4. 用户可关闭主动唤起

### 验收标准

- 主动提示低频、自然、不惊扰
- 用户能感受到“它在替我记着”
- 不会像营销通知或社交机器人骚扰

---

## 8. V3.0 Multi-Entry Companion System

### 产品定义

在 App 稳定后，再扩展外部入口。

### 目标

让温伴不只存在于 App 内，也能从外部入口轻触达。

### 入口方向

- QQ bot
- 其他消息入口
- 小程序或桌面端补入口

### 角色定义

外部入口只做：

- 轻聊天
- 唤起提醒
- 回到 App 的跳板

不做：

- 主照片库管理
- 主长期记忆管理
- 主图谱浏览

### 验收标准

- 用户能从外部入口回到温伴
- 但产品主阵地仍然是 App

---

## 9. 语音路线如何放进新版本

语音不单独作为“炫技模块”，而是横跨多个版本的底层输入能力。

### V1.0 - V1.1

- 浏览器语音可用
- 基本输入闭环可跑

### V1.5

- 手机端真实语音权限
- 更符合手持交互

### V2.0

- 老人语音评测集
- 方言 / 口语 / 自我修正测试
- 模型对比：
  - 浏览器 ASR
  - FunASR
  - Whisper

### V2.5 以后

- 围绕老人场景做更稳的专门优化

## 10. 版本与阶段的映射关系

为了避免再次出现“Stage 做完了但不知道属于哪个版本”的问题，之后每次开发都必须同时标明：

- 产品版本
- 对应阶段
- 本次目标

例如：

- `V1.0 / Stage A1`：Onboarding 与输入重构
- `V1.0 / Stage A2`：忆光照片主入口
- `V1.1 / Stage B1`：照片库基础页
- `V1.1 / Stage B2`：聊天触发照片召回

也就是说：

之后的 stage 只作为版本内分解，不再独立漂浮。

## 11. 现在建议立即执行的阶段

基于当前代码，建议马上进入下面这一组：

## `V1.0 Web Alpha`

### Stage A1：首次使用与信息架构重构

要做：

- 第一次使用 onboarding
- 设置页简化
- 用户卡片 / 温伴卡片拆清

### Stage A2：忆光输入重构

要做：

- 说一句
- 放一张照片
- 从照片开始写

### Stage A3：图谱页收口

要做：

- 只保留整理视图
- 不再承担主输入角色

### Stage A4：页面与交互压缩

要做：

- 按钮更少
- 文案更短
- 展示内容更多
- 去掉开发者感提示

---

## 12. 结论

接下来不应该再问：

- “还要不要继续做某个独立模块”

而应该统一问：

- “这一步是在推进哪个产品版本？”

这份路线图下，当前最明确的答案是：

**先把当前网页原型收成 `V1.0 Web Alpha`，再谈照片库、手机壳和真正 App 化。**
