# Stage 14 最小数据安全与用户方案

这一阶段不做完整商业化安全体系，只做当前大创原型必须讲清楚的边界、风险和后续落地方向。

## 一、当前版本定位

当前版本明确定位为：

- 单用户原型
- 本地演示版本
- 文件型持久化系统
- 适合组内测试与答辩演示

当前版本不应被描述为：

- 已完成多用户产品
- 已具备正式隐私合规能力
- 已完成商用部署安全方案

## 二、当前系统保存了什么

当前原型会在本地服务端保存：

- 聊天记录
- 候选记忆
- 草稿状态
- 活跃事件
- 修订日志
- 长期事实层
- reply trace
- 上传照片
- 温伴形象图

这些数据都属于需要谨慎对待的个人内容，因为它们可能包含：

- 人名
- 时间线
- 生活习惯
- 情绪表达
- 回忆片段
- 个人照片

## 三、这一阶段补的最小控制

### 1. 仓库层

- `.env` 不进仓库
- `data/` 不进仓库
- `uploads/` 不进仓库
- `.DS_Store` 不进仓库

### 2. 服务端层

- 关闭 `x-powered-by`
- 增加原型版 CORS 边界
- 图片上传保留大小限制
- 提供 `/api/privacy/summary` 作为演示说明接口

### 3. 产品口径层

需要明确对老师、组员和测试者说明：

- 当前是单用户原型
- 数据默认落在本机目录
- 当前没有账号体系
- 当前没有多用户隔离
- 当前没有正式加密存储

## 四、当前仍然没有完成的部分

这些部分属于后续 app 化必须补的正式工程内容：

- 用户注册 / 登录
- 用户级会话隔离
- 权限控制
- 数据删除与导出
- 服务端数据库
- 加密存储
- 访问日志与异常审计
- 线上部署与备份策略

## 五、如果后面要整合成 App，最少还要补什么

### 用户相关

至少补：

- 用户 ID
- 每个用户独立的数据空间
- 登录态
- 设备切换后的状态恢复

### 数据安全相关

至少补：

- 敏感字段数据库化
- 服务端鉴权
- 上传文件访问控制
- 删除账号时的级联删除
- 隐私说明与用户授权

### 产品交互相关

至少补：

- 首次使用隐私提示
- 用户可查看当前保存了哪些内容
- 用户可手动删除聊天 / 记忆 / 照片
- 用户知道哪些内容会长期保留

## 六、答辩时推荐表述

可以直接这样说：

> 当前版本是单用户本地原型，重点验证 AI 对话、记忆整理、照片墙和数字人形象这些核心体验是否成立。  
> 在数据层面，我们已经把敏感内容从纯前端缓存下沉到本地服务端文件，并补了最小的仓库隔离与原型级接口边界。  
> 但多用户隔离、正式鉴权、数据库与加密存储，属于下一阶段 App 化时必须完成的工程任务，目前没有把资源过早投入到这部分。

## 七、验收标准

- 项目文档不再回避用户与数据安全问题
- 组员知道当前原型保存了哪些数据
- 老师能清楚听到“当前版本做到哪里、没做到哪里”
- 不会把原型误说成正式商用后端
